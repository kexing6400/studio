## 🔄 循环时序

### 层级一：核心心跳 (每5分钟)

| 时间点 | 执行内容 | 负责 |
|--------|----------|------|
| :00 | 进程健康检查 + 任务流转 | 尚书省 |
| :05 | GitHub trending 扫描 | 客部 |
| :10 | 新项目机会分析 | 中书省 |
| :15 | 任务进度更新 | 太子 |
| :20 | 质量检查 (CI状态) | 刑部 |
| :25 | 资源清理 (logs, tmp) | 户部 |
| :30 | 推广状态检查 | 兵部 |
| :35 | 文档同步检查 | 吏部 |
| :40 | 风险扫描 (blocked任务) | 门下省 |
| :45 | 用户反馈收集 | 太子 |
| :50 | 系统自检报告 | 尚书省 |
| :55 | 休眠准备 | 各部 |

### 层级二：深度研究 (每30分钟)

```
客部(市场调研) 
    ├── GitHub Trending 扫描
    ├── 竞品分析
    ├── 技术栈趋势
    └── 机会点识别
```

### 层级三：开发执行 (持续)

```
工部 + 礼部 + 户部 + 刑部 并行执行
    ├── 代码开发
    ├── CI/CD 执行
    ├── 测试运行
    └── 部署发布
```

### 层级四：每日运营 (每天 08:00, 20:00)

```
太子 + 尚书省
    ├── 早间简报 (08:00)
    ├── 晚间总结 (20:00)
    ├── GitHub Stars 增长报告
    └── 次日计划
```

## 🎛️ 质量门禁 (Quality Gates)

每个任务必须通过以下关卡：

| 阶段 | 检查项 | 执行者 |
|------|--------|--------|
| Zhongshu | 技术方案完整性 | 门下省 |
| Menxia | 安全风险评估 | 刑部 |
| Doing | 代码规范检查 | ESLint |
| Doing | 单元测试覆盖率 ≥80% | Vitest/Jest |
| Doing | Docker 构建成功 |户部 |
| Review | CI/CD 全部通过 | 刑部 |
| Review | 文档完整性 | 吏部 |
| Done | 性能基准测试 | 工部 |

## 🔔 告警机制

| 级别 | 触发条件 | 处理 |
|------|----------|------|
| 🔴 紧急 | 进程崩溃 / API 限流 | 立即重启 + 通知 |
| 🟡 警告 | 任务 blocked >30min | 自动重试 + 标记 |
| 🟢 正常 | 所有检查通过 | 静默记录 |

## 📈 成功指标

- **吞吐量**: 每天完成 ≥1 个项目从 idea → deploy
- **质量**: CI/CD 100% pass rate
- **响应**: 旨意处理延迟 <30 分钟
- **透明度**: 所有状态变化有日志
